# Skill Architecture — Context Pack

## Overview

Single skill with three flows, triggered by natural language. No separate commands.

## Flows

### 1. Create (first time)

```
User: "create a context pack for this repo"
Agent: assess → scaffold → fill → seal → self-test → commit
```

Duration target: 5-10 min for ~200 file repo.

### 2. Update (agent PR)

```
Agent finishes work → preps PR → auto-patches .agent-context → separate commit in PR
```

Trigger: implicit — part of the agent's PR workflow.
Guard: only when the agent did the work. Never on human PRs.

### 3. Update (manual catchup)

```
User: "update the context pack"
Agent: diff since last seal → propose patches → human approves each → apply → seal
```

Duration target: 2-5 min depending on diff size.

## Dependencies

- `chorus` CLI v0.9.0+ (npm `agent-chorus`)
- Git repository
- Agent with file read/write permissions

## Self-test protocol (first-time create only)

1. Generate 3 questions:
   - L: "Where is <specific value> defined?" (lookup)
   - M: "What files must change to <specific change>?" (impact analysis)
   - H: "How would you diagnose <specific symptom>?" (diagnosis)

2. For each question, write ground truth from the agent's repo knowledge

3. Evaluate: does the pack content answer each question?
   - If yes: pack section is strong
   - If no: identify which section is weak, improve it

4. This is a mental evaluation, not a full sub-agent experiment run.
   The full experiment framework exists in `tests/behaviour/` for repos
   that want rigorous validation.

## Update detection (for agent PR flow)

The agent maps its own changes to context pack sections:

| Change type | Affected sections |
|---|---|
| New file in existing directory | CODE_MAP (maybe), search_scope search_directories (no change) |
| New file in new directory | CODE_MAP, search_scope search_directories |
| Modified file with line changes | search_scope verification_shortcuts (line numbers) |
| Deleted file | CODE_MAP, completeness_contract, search_scope |
| New dependency | OPERATIONS (validation commands) |
| New command/API | CODE_MAP, SYSTEM_OVERVIEW command table, INVARIANTS checklist |
| Schema change | INVARIANTS checklist, completeness_contract |

## Catchup detection (for manual update flow)

```bash
# Files changed since last seal
git diff $(git log -1 --before="$(jq -r '.generated_at' .agent-context/current/manifest.json)" --format=%H)..HEAD --name-only -- . ':!.agent-context'
```

The agent reads each changed file, determines what kind of change it is, and proposes the minimal context pack edit.

## Testing plan

1. **Test on a new work repo** (Edelman, different type from stream-models and agent-chorus)
   - Use the skill to create a context pack from scratch
   - Validate the output is seal-able and useful
   - Run a lightweight experiment (2-3 tasks, bare vs pack)

2. **Test the update flow**
   - Make a code change via agent
   - Verify the agent auto-preps .agent-context updates in the PR
   - Verify changes are in a separate commit

3. **Test the catchup flow**
   - Manually make changes, commit without agent
   - Ask agent to update the context pack
   - Verify it proposes correct patches and doesn't overwrite unrelated sections
